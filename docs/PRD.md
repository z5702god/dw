> Version: 1.0

> Status: Draft

## 1. 產品概述

### 1.1 一句話描述

一款情侶共用的 iOS 飲食紀錄 App，集結飲食日誌、美食地圖、吃飯打卡與點數獎勵系統，讓異地情侶能共享美食體驗並互相督促按時吃飯。

### 1.2 背景與動機

- 兩人分別在 **台北** 和 **台中**，各自發現好吃的餐廳時缺乏共同紀錄的工具
- 希望將美食心得結合地圖，下次見面時可以一起去吃
- 需要一個溫暖但有效的方式督促女朋友按時吃飯
- 市面上的飲食紀錄 App 缺乏「情侶共享」和「互動激勵」的功能

### 1.3 目標用戶

| 角色 | 描述 |
|------|------|
| **用戶 A（Luke）** | 在台北，負責督促對方吃飯、共同紀錄美食 |
| **用戶 B（女朋友）** | 在台中，被提醒按時吃飯、共同紀錄美食 |

> 本產品 V1 僅供兩人使用，不開放註冊。

---

## 2. 產品目標與成功指標

### 2.1 核心目標

1. **共享美食紀錄** — 兩人的飲食心得即時同步，建立共同的美食回憶
2. **美食地圖探索** — 以地圖形式呈現去過的餐廳，方便未來規劃約會
3. **督促按時吃飯** — 透過遊戲化機制（打卡 + 點數 + 獎勵）讓按時吃飯變成有趣的事

### 2.2 成功指標（V1）

| 指標 | 目標 |
|------|------|
| 每週新增飲食紀錄數 | >= 10 則（兩人合計） |
| 女朋友每日打卡率 | >= 80%（至少 2 餐/天） |
| 打卡連續天數 | >= 14 天 |
| 獎勵兌換頻率 | 每月至少 1 次 |

### 3.1 Feature 1: 用戶驗證（Auth）

**優先級：P0（必要）**

| 項目 | 說明 |


### 3.2 Feature 2: 飲食紀錄 / 日誌（Timeline）

**優先級：P0（必要）**

#### 3.2.1 紀錄內容

| 欄位 | 類型 | 必填 | 說明 |
|------|------|------|------|
| 餐廳名稱 | 文字 | 是 | 餐廳或店家名稱 |
| 照片 | 圖片（最多 5 張） | 否 | 從相簿選取或拍照 |
| 心得 | 長文字 | 是 | 飲食感想、推薦菜色等 |
| 推薦評價 | 列舉 | 是 | 推薦 / 普通 / 踩雷 |
| 城市 | 列舉 | 是 | 台北 / 台中 |
| 位置座標 | 經緯度 | 自動 | 根據城市預設，或由用戶微調 |
| 紀錄者 | 用戶 ID | 自動 | 當前登入用戶 |
| 建立時間 | 時間戳 | 自動 | 伺服器時間 |

#### 3.2.2 時間軸列表

- 顯示兩人所有飲食紀錄，按時間倒序排列
- 每則紀錄顯示：縮圖、餐廳名稱、心得摘要（前 2 行）、評價標籤、城市、日期
- 頂部提供篩選器：全部 / 推薦 / 普通 / 踩雷
- 點擊進入詳細頁面
- 支援左滑刪除（僅可刪除自己的紀錄）

#### 3.2.3 詳細頁面

- 照片輪播（支援左右滑動）
- 餐廳名稱 + 評價標籤
- 城市 + 日期
- 完整心得文字

#### 3.2.4 新增紀錄

- 表單包含：餐廳名稱、照片選擇器、城市選擇、評價選擇（Segmented）、心得文字框
- 照片上傳時壓縮至 JPEG 品質 0.7、最大邊 1200px
- 儲存時顯示上傳進度
- 儲存成功後自動關閉表單

### 3.3 Feature 3: 美食地圖（Food Map）

**優先級：P0（必要）**

#### 3.3.1 地圖顯示

- 使用 Apple MapKit（SwiftUI Map API，iOS 17+）
- 每筆飲食紀錄在地圖上以標記（Annotation）顯示
- 標記顏色依評價區分：
  - 推薦 → 綠色
  - 普通 → 橘色
  - 踩雷 → 紅色

#### 3.3.2 城市切換

- 頂部提供 Segmented Picker：台北 / 台中
- 切換時地圖自動移動至對應城市的預設中心點
- 預設座標：
  - 台北：25.033, 121.565
  - 台中：24.147, 120.673

#### 3.3.3 篩選功能

- 可按評價篩選：全部 / 推薦 / 普通 / 踩雷
- 篩選器以水平滑動的 Chip 呈現，覆蓋在地圖上方

#### 3.3.4 互動

- 點擊地圖標記 → 彈出半頁 Sheet 顯示該餐廳的詳細紀錄
- Sheet 支援拖動展開至全螢幕

### 3.4 Feature 4: 吃飯打卡與點數系統（Check-In & Points）

**優先級：P0（必要）**

#### 3.4.1 打卡機制

| 規則 | 說明 |
|------|------|
| 打卡時段 | 早餐 / 午餐 / 晚餐 |
| 每日打卡次數 | 每個時段最多 1 次，每日最多 3 次 |
| 準時判定 | 在設定時間的前後 **30 分鐘** 內打卡算準時 |
| 準時獎勵 | 10 點 |
| 遲到獎勵 | 3 點（鼓勵至少有吃） |
| 重複打卡 | 同一時段不可重複打卡，顯示「已打卡」狀態 |

#### 3.4.2 打卡頁面

- 頂部顯示當前總點數（大字體、醒目）
- 今日已獲得點數
- 三個打卡卡片（早餐 / 午餐 / 晚餐），每個顯示：
  - 餐別圖示 + 名稱
  - 對方是否已打卡（即時同步）
  - 打卡按鈕 或 已打卡狀態
- 打卡成功後彈出 Alert：
  - 準時：「準時吃飯！獲得 10 點」
  - 遲到：「雖然遲了，但至少有吃！獲得 3 點」

#### 3.4.3 對方打卡可見性

- 可以即時看到對方今天哪些餐已打卡
- 目的：讓你知道女朋友有沒有按時吃飯

### 3.5 Feature 5: 獎勵系統（Rewards）

**優先級：P0（必要）**

#### 3.5.1 獎勵清單

- 由兩人自訂獎勵項目
- 每個獎勵包含：名稱 + 所需點數
- 任一用戶都可以新增獎勵
- 支援左滑刪除

#### 3.5.2 獎勵兌換

| 規則 | 說明 |
|------|------|
| 兌換條件 | 當前點數 >= 獎勵所需點數 |
| 兌換動作 | 扣除對應點數，獎勵標記為「已兌換」|
| 點數不足 | 兌換按鈕灰色不可點擊 |

#### 3.5.3 獎勵頁面

- 頂部顯示可用點數
- 可兌換獎勵列表（每列：獎勵名稱、所需點數、兌換按鈕）
- 已兌換獎勵列表（刪除線顯示）
- 右上角 + 按鈕新增獎勵

#### 3.5.4 新增獎勵表單

- 獎勵名稱（文字輸入）
- 所需點數（Stepper，10-1000，步進 10）
- 快速預設按鈕：30 / 50 / 100 / 200

### 5.3 安全規則

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // 情侶空間：只有配對的兩人可以存取
    match /couples/{coupleId}/{document=**} {
      allow read, write: if request.auth != null
        && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.coupleId == coupleId;
    }
    // 用戶資料：任何已登入用戶可讀，只有本人可寫
    match /users/{userId} {
      allow read: if request.auth != null;
      allow write: if request.auth.uid == userId;
    }
  }
}
```

## 6. 非功能需求

### 6.1 效能

| 項目 | 目標 |
|------|------|
| App 啟動時間 | < 2 秒（含 Firebase 初始化）|
| 照片上傳 | 壓縮後單張 < 500KB，5 張總計 < 5 秒（Wi-Fi）|
| 資料同步 | Firestore 即時同步，延遲 < 1 秒 |
| 地圖載入 | < 1 秒顯示所有標記（預期 < 200 筆紀錄）|

### 6.2 離線支援

- Firestore 內建離線快取，斷網時可瀏覽已載入的紀錄
- 離線新增的紀錄在恢復連線後自動同步
- 離線打卡在恢復連線後自動上傳

### 6.3 隱私與安全

- 所有資料僅限配對的兩人存取（Firestore Security Rules）
- 照片存放在 Firebase Storage，僅透過下載 URL 存取
- 不收集任何第三方分析數據（V1）
- 不需要位置權限（城市座標為手動選擇）

### 6.4 可靠性

- Firebase 99.95% SLA
- 資料自動備份（Firestore 每日自動備份可在 Console 設定）

## 9. 風險與限制

| 風險 | 影響 | 緩解措施 |

| 資源 | 每日免費額度 |
|------|-------------|
| 文件讀取 | 50,000 次 |

### 10.2 名詞解釋

| 名詞 | 定義 |
|------|------|
| 情侶空間（Couple Space）| Firestore 中 `couples/{coupleId}` 下的所有資料，兩人共享 |
